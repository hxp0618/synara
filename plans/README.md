## Dependency notes

- 001 must land first because short timers and unattended work magnify existing lifecycle gaps: orphaned dispatch, weak cancellation, heartbeat collisions, stale waiting states, and dispatch-failure stop-on-error behavior.
- 002 depends on 001 because one-shot/short timers need durable run refs, recovery, and safe scheduler dispatch before they can run reliably.
- 003 depends on 002 because the creation UX must understand `once` schedules, next-run preview, and short-timer warnings.
- 004 depends on 001 because triage actions such as cancel/archive/read must not race provider completion or leave active work running.
- 005 is intentionally last because cron/timezone/policies have higher regression risk and should build on the proven timer + triage foundation.

## Quality and performance guardrails

- Keep scheduler polling adaptive. Do not introduce a global 1-second loop unless explicitly measured and accepted.
- Keep recurrence math isolated in server-side schedule helpers with focused tests.
- Preserve existing automation rows; new schedule fields must decode old JSON safely.
- Use best-effort side effects for non-critical UI notes/logging. A saved automation must not be reported as failed because optional activity logging failed.
- Avoid coupling chat composer parsing directly to persistence. Parser → draft → confirmation → API create should be the long-term shape.
- Prefer conservative triage classification. Unknown/unread is safer than auto-archiving a run with hidden findings.
- Do not silently persist or promise attachments in scheduled runs until server/provider contracts support them.

## Findings considered and rejected

Automation/timer run (2026-06-19):

- Implementing full cron before one-shot timers: rejected for now because it adds timezone/DST complexity before the scheduler lifecycle is hardened.
- Persisting composer attachments into scheduled automation runs immediately: rejected for now because current chat automation creation sends prompt-only automation input; attachment persistence needs separate contract/provider design.
- Using a constant sub-second scheduler loop: rejected for performance and SQLite churn; use adaptive wake-ups instead.
- Letting generated intent create automations without confirmation: rejected for unattended background-work safety; use an editable draft for risky or ambiguous schedules.
